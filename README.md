## 1. 项目简介

### 1.1 什么是 Flux

Flux 是一个创新性的 **Minecraft 去中心化联机方案**。

传统 Minecraft 联机需要一台高性能服务器运行完整的游戏服务端（Java Edition Server），消耗大量 CPU 和内存资源。Flux 的思路完全不同：

服务端不运行游戏引擎，不模拟世界，只负责：
- 验证每个玩家的操作是否合法
- 在玩家之间转发合法的操作
- 记录事件日志

### 1.2 与传统方案的对比

| 指标         | 官方 Java 服务端  | Flux         |
| ---------- | ------------ | ------------ |
| **内存占用**   | 2-4 GB       | < 50 MB      |
| **CPU 使用** | 单核满载（游戏逻辑）   | 几乎为零（仅校验和转发） |
| **硬件成本**   | 高性能服务器 / 云主机 | 任意电脑 / 树莓派   |
| **最大玩家数**  | 受限于服务器算力     | 理论无上限        |
| **游戏体验**   | 取决于服务器延迟     | 取决于玩家本地性能    |
| **世界归属**   | 服务端拥有世界      | 玩家共同拥有世界     |

### 1.3 核心优势

- **极低资源消耗**：服务端只做数据包校验和转发，不需要运行游戏引擎
- **去中心化**：没有"管理员"可以作弊，规则由代码强制执行
- **低成本**：任意一台电脑即可运行，不需要高配服务器
- **可审计**：所有事件记录在日志中，哈希链防篡改，可追溯任何操作
- **隐私保护**：端到端加密，服务端看不到玩家的明文数据

## 2. 设计哲学

### 2.1 服务端是审计，不是世界

传统 Minecraft 服务端的职责：

```
物理引擎 + AI 寻路 + 世界生成 + 区块管理 + 网络通信 + 反作弊
```

Flux 服务端的职责：

```
加密/解密 + 数据包合法性校验 + 广播转发 + 事件日志
```

服务端不知道"钻石矿长什么样"，不知道"僵尸怎么走路"，它只知道：
- 这个玩家移动速度是否超过 10 m/s？
- 这个玩家攻击距离是否超过 4.5 格？
- 这个方块操作是否符合因果关系？

### 2.2 四条规则

Flux 的防作弊不依赖复杂的游戏逻辑，只检查四条不可违反的物理边界：

1. **速度不能超限** — 正常疾跑 ~5.6 m/s，硬上限 10 m/s
2. **距离不能超标** — 攻击距离 > 4.5 格、交互距离 > 6 格 → 直接拒绝
3. **物品必须守恒** — 从箱子里拿东西，箱子里要真的有
4. **事件必须有因** — 说挖了钻石矿，坐标上要有钻石矿

### 2.3 影子见证人

当一个区块只有单个玩家时，他可以肆无忌惮地修改客户端。解决办法：

> 只要附近有第二个玩家 B，A 发送的每一个涉及实体的数据包，不仅服务端自己校验，还会转发给 B。B 的客户端在后台并行运行相同规则，如果 B 的计算结果和 A 的不一致，服务端就触发仲裁。

### 3.2 数据流

```
玩家操作（移动/破坏/放置/攻击）
    │
    ▼
Fabric Mod 拦截事件
    │
    ▼
编码为二进制协议包（小端序，紧凑格式）
    │
    ▼
AES-128-GCM 加密（ECDH 协商的会话密钥）
    │
    ▼
TCP 发送到 Flux 服务端
    │
    ▼
服务端解密 + GCM 标签验证 + SeqID 防重放
    │
    ▼
包校验器检查四条铁律
    │
    ├─ 合法 → 记录事件日志 → 加密广播给其他玩家
    │
    └─ 非法 → 丢弃 + 回滚/冻结/踢出
```

### 3.3 启动时序

```
服务端启动
    │
    ├─ [1] 生成 ECDH 密钥对
    ├─ [2] 加载事件日志 + 世界种子
    ├─ [3] 启动 TCP 监听
    └─ [4] 等待客户端连接

客户端连接
    │
    ├─ [1] TCP 连接建立
    ├─ [2] CLIENT_HELLO → 客户端发送 ECDH 公钥
    ├─ [3] SERVER_HELLO → 服务端发送 ECDH 公钥
    ├─ [4] 双方独立派生 AES-128 会话密钥
    ├─ [5] AUTH_REQUEST → 客户端发送 UUID + 用户名
    ├─ [6] AUTH_SUCCESS → 认证通过
    ├─ [7] WORLD_SEED → 服务端下发世界种子
    ├─ [8] EVENT_LOG_ENTRY × N → 回放历史事件
    ├─ [9] SYNC_COMPLETE → 同步完成
    └─ [10] 进入 ACTIVE 状态，开始游戏
```

### 4.2 加密引擎 (crypto_engine)

**职责**：密钥协商 + 数据加解密

**算法**：
- 密钥协商：ECDH (X25519 / Curve25519)
- 对称加密：AES-128-GCM（认证加密）
- 密钥派生：SHA-256(shared_secret) → 取前 16 字节

**流程**：
```
双方各自生成临时密钥对
    → 交换公钥
    → 各自计算共享密钥（结果相同）
    → SHA-256 哈希取前 16 字节 → AES-128 会话密钥
    → 所有后续通信使用此密钥加密
    → 断开连接 → 密钥销毁（仅存于内存，不落盘）
```

**安全特性**：
- 前向安全：每次连接使用全新临时密钥，即使长期密钥泄露也无法解密历史通信
- 认证加密：AES-GCM 同时提供保密性和完整性，任何篡改都会被检测
- 防重放：每个包携带递增的 SeqID，拒绝重复或乱序的包

### 4.3 二进制协议层 (protocol)

**职责**：定义所有数据包的格式、编解码

- 极致紧凑：移动包仅 59 字节

- 强类型：每种包类型有固定的载荷结构

### 4.4 包校验器 (packet_validator)

**职责**：执行四条防作弊规则

**实现**：
- 维护每个玩家的位置状态（用于速度计算）
- 维护方块世界状态（用于因果关系校验）
- 维护容器槽位状态（用于物品守恒校验）
- 连续违规计数 → 自动升级惩罚

**惩罚等级**：

| 等级 | 条件 | 动作 | 游戏内表现 |
|------|------|------|------------|
| Lv.1 | 单次偶发超限 | 丢包 + 回滚 | 玩家被"拉回" |
| Lv.2 | 连续 3 次篡改 | 冻结客户端 | 世界静止，无法交互 |
| Lv.3 | 累计 5 次或畸形包 | 踢出 | 弹出"连接被关闭" |

### 4.5 事件日志引擎 (event_logger)

**职责**：记录所有游戏事件，支持世界同步和审计

**存储格式**：
```
事件条目 = 时间戳(4B) + 客户端ID(1B) + 事件类型(1B) + 数据长度(2B) + 数据(NB) + SHA-256哈希(32B)
```

**哈希链防篡改**：
```
事件 #0 的 hash = SHA-256(全零 || 事件 #0 数据)
事件 #1 的 hash = SHA-256(事件 #0 的 hash || 事件 #1 数据)
事件 #2 的 hash = SHA-256(事件 #1 的 hash || 事件 #2 数据)
...
```

任何一条事件被篡改，后续所有哈希都会变化，可被监控面板检测。

**文件管理**：
```
Flux_data/
├── seed.bin              ← 世界种子 (int64)
└── events/
    ├── events_0000.bin   ← 事件日志分片 #0 (最大 4MB)
    ├── events_0001.bin   ← 事件日志分片 #1
    └── ...
```

### 4.6 中继引擎 (relay_engine)

**职责**：客户端管理 + 区块所有权 + 盲转发广播

**客户端状态机**：
```
DISCONNECTED → HANDSHAKE → AUTHENTICATED → SYNCING → ACTIVE
                                    ↓              ↓
                              AUTH_FAIL       FROZEN → KICKED
```

**区块所有权**：
- 区块大小：16×16 格
- 谁距离区块中心最近，谁就是"托管人"
- 30 秒无更新 → 自动释放所有权
- 新玩家靠近 → 重新分配

**盲转发广播**：
```python
for 每个活跃客户端:
    if 不是发送者 and 不是冻结状态:
        用该客户端的会话密钥加密后发送
```

每个客户端使用独立的会话密钥，服务端无法伪造其他客户端的数据。

### 4.7 TCP 服务端 (server)

**职责**：接受 TCP 连接，驱动整个服务端流程

**技术栈**：
- Python asyncio 异步 I/O
- 每个客户端一个协程处理
- 独立的维护协程（超时检测、区块 GC）
- 独立的心跳协程（定期 PING）

## 5. 二进制协议规范

### 5.1 线格式 (Wire Format)

```
+----------+--------+-------------+------------------+----------------+----------+----------+
| Magic(2B)| Type(1B)| Length(4B)  |    IV (12B)      | SeqID (4B)     | Data(NB) | Tag(16B) |
+----------+--------+-------------+------------------+----------------+----------+----------+
|<------------ Header (7B) ------>|<---------------- 加密载荷 (Length 字节) ----------------->|
```

- **Magic**: `0x484E` ("HN")，用于在字节流中定位包头
- **Type**: 数据包类型（见上表）
- **Length**: IV + SeqID + Data + Tag 的总字节数
- **IV**: 随机初始化向量，保证相同数据每次加密结果不同
- **SeqID**: 递增序列号，防重放
- **Data**: 实际游戏数据
- **GCM Tag**: 16 字节认证标签，任何篡改都会导致校验失败
- **字节序**: 小端序 (Little-Endian)

### 5.2 包大小对比

| 包类型  | Flux | 官方协议   | 节省  |
| ---- | ---- | ------ | --- |
| 玩家移动 | 59 B | ~200 B | 70% |
| 方块破坏 | 52 B | ~150 B | 65% |
| 方块放置 | 64 B | ~250 B | 74% |

## 6. 加密与安全机制

### 6.1 三根支柱

| 支柱 | 算法 | 解决的问题 |
|------|------|-----------|
| 动态密钥 | ECDH (X25519) | "我们是谁" — 每次连接全新临时密钥 |
| 流加密 | AES-128-GCM | "数据没被看且没被改" — 认证加密 |
| 防重放 | SeqID 单调递增 | "这句话你刚才已经说过了" |

### 6.2 密钥生命周期

```
连接建立 → 生成临时 ECDH 密钥对
    → 交换公钥（明文传输）
    → 各自计算共享密钥
    → SHA-256(shared_secret) → 取前 16 字节 → AES-128 会话密钥
    → 所有后续通信加密
    → 断开连接 → 密钥销毁（仅存于内存，不落盘）
```

### 6.3 安全保障

- **前向安全**：每次连接使用全新临时密钥
- **零知识**：服务端无法解密玩家的明文数据（只知道数据是否合法）
- **防中间人**：GCM 标签验证，任何篡改都会被检测
- **防重放**：SeqID 严格递增，拒绝重复包

## 7. 防作弊规则

### 7.1 规则一：速度不能超限

```
正常疾跑:      ~5.6 m/s
鞘翅飞行:      ~8.0 m/s
蓝冰船:        ~7.0 m/s
药水加速疾跑:  ~8.0 m/s
硬上限:        10.0 m/s（超过就是异常）
```

### 7.2 规则二：距离不能超标

```
攻击距离: 最大 4.5 格（锋利 III 剑 + 力量 II 的极限）
交互距离: 最大 6 格（开箱子、按按钮等）
超过 → 数据包直接丢弃
```

### 7.3 规则三：物品必须守恒

```
合成: 材料消耗 = 产品产出
捡拾: 地面掉落物存在 → 物品栏增加
箱子操作: 箱子里真的有这些东西
事件日志是最终账本
```

### 7.4 规则四：事件必须有因

```
挖方块: 该坐标确实有方块
放方块: 该坐标确实是空气
开箱子: 该坐标确实有箱子
没有历史记录 = 假账
```

## 9. 数据流详解

### 9.1 玩家移动

```
玩家移动
    → Fabric Mod 每 50ms 检测位置变化
    → 编码: seq_id(4B) + x(4B) + y(4B) + z(4B) + yaw(4B) + pitch(4B) + on_ground(1B) = 25B
    → AES-128-GCM 加密
    → TCP 发送
    → 服务端解密
    → 速度校验 (distance/time_delta < 10.0 m/s)
    → 合法 → 更新区块所有权 → 广播给其他玩家
    → 非法 → 回滚位置
```

### 9.2 方块破坏

```
玩家破坏方块
    → Fabric Mod 拦截 AttackBlock 事件
    → 编码: seq_id(4B) + x(4B) + y(4B) + z(4B) + block_id(1B) + face(1B) = 18B
    → AES-128-GCM 加密
    → TCP 发送
    → 服务端解密
    → 距离校验 (< 6.0 格)
    → 因果校验 (该坐标确实有方块)
    → 合法 → 记录事件日志 → 广播给其他玩家
    → 非法 → 丢弃
```

### 9.3 新玩家加入

```
新玩家 TCP 连接
    → ECDH 密钥协商
    → 身份认证
    → 接收世界种子 (WORLD_SEED)
    → 在本地生成原始地形
    → 接收历史事件 (EVENT_LOG_ENTRY × N)
    → 逐条回放方块变更
    → 接收同步完成信号 (SYNC_COMPLETE)
    → 正式进入世界
```

## 13. FAQ

### Q: Flux和 Paper/Spigot 有什么区别？

Paper/Spigot 是官方服务端的优化分支，仍然需要运行完整游戏引擎。Flux 完全不运行游戏引擎，只做数据包校验和转发。

### Q: 玩家可以作弊吗？

Flux 检查四条不可违反的物理边界。客户端可以尝试发送假数据，但：
- 速度超限 → 丢包
- 距离超标 → 丢包
- 物品凭空出现 → 丢包 + 冻结
- 方块不存在 → 丢包

### Q: 新玩家加入时怎么还原世界？

三阶段同步：种子生成原始地形 → 回放事件日志 → 获取动态实体快照。

### Q: 服务端能看到玩家的数据吗？

不能。所有数据使用 AES-128-GCM 端到端加密，服务端只能看到密文。服务端只知道"这个包是否合法"，不知道具体内容。

### Q: 支持多少玩家？

理论上无上限。服务端只做校验和转发，不运行游戏逻辑。实际限制取决于网络带宽和 TCP 连接数。

### Q: 可以用手机/平板玩吗？

目前只支持 Minecraft Java Edition + Fabric Mod。Bedrock 版本需要不同的客户端实现。

### Q: 断线重连后世界会丢失吗？

不会。所有事件都记录在服务端的日志中，重连后会重新同步。
